@AGENTS.md

Urvar CRM — internal CRM for Urvar Natural Pvt Ltd (organic fertilizers/bio-fertilizers B2B/B2C). See [README.md](README.md) for tech stack, setup, demo accounts, and project structure — don't duplicate that here.

## Repo-specific conventions

- **Prisma 7, custom client output.** The generator (`prisma/schema.prisma`) writes to `src/generated/prisma` (gitignored), not `node_modules/.prisma`. Import enums/types from `@/generated/prisma/...` (e.g. `@/generated/prisma/enums`), not `@prisma/client`. Run `npm run db:generate` after editing the schema.
- **Datasource URL lives in `prisma.config.ts`**, not the `datasource` block in `schema.prisma` — it still reads `DATABASE_URL` from `.env`.
- **RBAC is enforced in application code, not middleware.** `middleware.ts` only redirects unauthenticated requests to `/login`. Every Server Action / route handler that touches CRM data must call `can()` / `assertCan()` and scope its Prisma `where` with `scopeWhere()` (all in `src/lib/permissions.ts`). Scope is one of `none` / `own` / `territory` / `all` per (role, module, action) — adding a module or action means updating the `PERMISSIONS` matrix for all 5 roles, not just the one being tested.
- **Phone numbers are stored as plain 10-digit Indian mobile numbers** (see `src/lib/validations/lead.ts`, `customer.ts`). Only convert to E.164 at the Twilio dial boundary via `toE164()` in `src/lib/phone.ts` — don't normalize on save.
- **Local file storage, not cloud.** Call recordings and other generated files live under `storage/` (gitignored; base path is `CALL_RECORDINGS_DIR` in `.env`).
- **Windows dev box, no Docker/WSL.** For local dev, Postgres is a portable, no-admin install, not a Windows service — run `npm run pg:start` after every reboot before `npm run dev`. Don't assume `psql`/`pg_ctl` are on PATH.
- **This same machine also runs production** (crm.urvarindia.com). `scripts/deploy-setup.ps1` (elevated, idempotent) registers three real Windows services with `StartType: Automatic` that survive reboots on their own — `UrvarCrmPostgres` (port 5432, binaries at `D:\PostgresPortable\pgsql\bin`), `Cloudflared` (tunnel routing `crm.urvarindia.com` → `localhost:3000`), and `UrvarCrmApp` (`next start`, wrapped via WinSW at `D:\Urvar-CRM-service`). After a reboot, check `Get-Service UrvarCrmPostgres, Cloudflared, UrvarCrmApp` before assuming anything needs manual restarting — don't conflate this with the dev-only `pg:start` flow above. A nightly `pg_dump` scheduled task (`UrvarCrmPgBackup`, 2 AM) backs up to `E:\UrvarCrmBackups` (last 14 kept). To ship a new build to prod: `npm run db:generate` / `db:migrate:deploy` / `npm run build`, then `Restart-Service UrvarCrmApp`.
- **`Restart-Service UrvarCrmApp` needs an elevated PowerShell window** — Claude Code's own shell isn't elevated, so it can run the build/migrate steps but the human operator has to do the actual restart (right-click PowerShell → "Run as Administrator"; a non-elevated window fails with "Cannot open ... service on computer '.'" even though `Get-Service` still works fine read-only).
- **No automated test suite.** Validate changes with `npm run lint` and `npm run build` (type-checks via `tsc`), plus manual testing through `npm run dev`.
- **Better-auth user email/password are edited directly via Prisma, not the admin plugin.** `src/app/(dashboard)/users/actions.ts` updates `User.email` directly and, for password resets, hashes the new password with `hashPassword` from `better-auth/crypto` and writes it to the user's `Account` row (`providerId: "credential"`, upserted) — this matches what better-auth's own `admin.setUserPassword` endpoint does internally, without pulling in the full `admin` plugin (which would require mapping our 5-role RBAC onto better-auth's own role/permission system). Never log the plaintext password to the audit trail — log a `passwordChanged: true` flag instead.
- **README's "Build Roadmap" section is stale.** It reads as in-progress, but every listed module (Customers, Products, Quotations, Reports, Audit logs) already exists under `src/app/(dashboard)/`. Confirm what's actually implemented by reading the code, not the roadmap.
- **The real Urvar logo lives at `public/urvar-icon.png`** (the leaf-mark icon, sourced from `urvarindia.com/icon.png`) — used in the sidebar, mobile nav drawer, and login page. It's mirrored at `src/app/icon.png` (Next.js app-icon convention) and baked into `src/app/favicon.ico` as a hand-built multi-size ICO (16/32/48px, generated via `sharp` since no PNG→ICO CLI was on this box — see git history for the one-off script). If Urvar ever issues a brand refresh, re-pull from `urvarindia.com` (their CDN 403s a bare `WebFetch`/no-UA request — use `curl` with a real browser `User-Agent` string) — the full vertical wordmark lockup is at `/logo.png` and `/logo.svg` there, distinct from the square icon.
- **There is no separate dev database.** `DATABASE_URL` in `.env` points at the same `urvar_crm` database on `localhost:5432` that production reads from — this box's dev environment and crm.urvarindia.com share one Postgres instance. Run `npx prisma migrate status` before `prisma migrate dev` to confirm there's no drift (a drift-triggered reset prompt would wipe production data); additive nullable-column migrations are low-risk but still real production schema changes, not throwaway dev ones.
- **Local dev on this box can't use port 3000 or a `.env.local` file.** Port 3000 is bound by the live `UrvarCrmApp` service. Run `next dev` on another port (e.g. `PORT=3001`). `.env`'s `NEXT_PUBLIC_APP_URL`/`BETTER_AUTH_URL` are hardcoded to `https://crm.urvarindia.com`, so better-auth's client will try to hit production unless overridden — pass `NEXT_PUBLIC_APP_URL`/`BETTER_AUTH_URL` as shell env vars on the `npm run dev` command itself (they take precedence over `.env`). Don't create a `.env.local` to do this instead: `UrvarCrmApp`'s WinSW service runs `next start` from this exact same `D:\Urvar-CRM` working directory, so a stray `.env.local` would silently override production's auth URLs on its next restart too.
- **Customers can be pinned to an exact map location.** `Customer.latitude`/`longitude` (nullable `Float`) are set via a Leaflet + OpenStreetMap dialog (`customers/[customerId]/location-dialog.tsx` + `location-map.tsx`, dynamically imported with `ssr: false`) — no API key/billing since it's OSM tiles, not Google Maps. Includes a browser-geolocation "Use My Location" button with an accuracy-radius circle for field reps pinning a site in person.
- **Calling works from both Leads and Customers.** `initiateCall`/`logCall` (`src/app/(dashboard)/calls/voice-actions.ts`, `actions.ts`) take a `{ leadId } | { customerId }` discriminated union, not a bare id — `completeVoiceCall` already branched on `existing.leadId`/`existing.customerId` before this, so the two entry points were generalized to match that existing precedent rather than duplicated per entity. The `Call` model already had both FKs from the start; the global `/calls` list joins both `lead` and `customer` and links to whichever is set.
